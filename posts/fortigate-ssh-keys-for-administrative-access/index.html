    FortiGate SSH keys for administrative access

    2022-09-22

    Introduction to SSH

    OpenSSH is the premier connectivity tool for remote login with the SSH protocol. It encrypts all traffic to eliminate eavesdropping, connection hijacking, and other attacks.

    That’s a cite from the official OpenSSH Webpage and also OpenSSH as for today is a very common type of connection to routers, switches, firewalls and so on. What we’re going to cover in this post is how to connect to a (FortiGate)[https://www.fortinet.com/] device as an administrative User via ssh(1)

    We are going to use ed25519 which was introduced to OpenSSH in version 6.5 and is based on the (Twisted Edwards Curve)[https://en.wikipedia.org/wiki/Twisted_Edwards_curve]. Compared to the very popular RSA key algorithm the ed25519 is a very good alternative since it’s faster and with only 68 characters much smaller.

    Create ssh key pair

    First we do have to create a ssh key pair.

    ~ % ssh-keygen -t ed25519 -f zigafrog
Generating public/private ed25519 key pair.
Enter passphrase (empty for no passphrase):
Enter same passphrase again:
Your identification has been saved in zigafrog
Your public key has been saved in zigafrog.pub
The key fingerprint is:
SHA256:to63M8jg42D0cyTx/NvE1EhkFOFK/AhO8stqUOZyzpM kainzjoh@nfeb-kainzjoh.local
The key's randomart image is:
+--[ED25519 256]--+
|         .*o     |
|       . +       |
|    o o o o      |
|    oO o = o     |
|  .+. * S + .    |
| .o.++ + +       |
|  oBo++.o o      |
| . .E+oo++       |
|   ooo..++.      |
+----[SHA256]-----+

    After issuing this command you’ll get zigafrog and zigafrog.pub in your current working directory. For further processing we’re going to move the files over to $HOME/.ssh

    ~ % mv zigafrog zigafrog.pub $HOME/.ssh/

    There is one very important thing to be mentioned here: Do not publish your private key

    After changing directories to $HOME/.ssh we can copy the newly generated public key to our clipboard

    .ssh % cat zigafrog.pub | pbcopy

    Transfer and Configure FortiGate

    Connect to your FortiGate Firewall using ssh and enter the admin credentials. For demonstration purposes we’re going to add this to our manager user. With set ssh-public-key1 you can define the key in the configuration. As of FortiOS-7.2 you can add up to three different ssh keys per user

    .ssh % ssh manager@10.16.2.254 
Warning: Permanently added '[10.16.2.254]:8022' (ED25519) to the list of known hosts.
manager@10.16.2.254's password:
fw # config system admin
fw (admin) # edit manager
fw (manager) # set ssh-public-key1 "ssh-ed25519 AAAAC3NzaC1lZDI1NTE5AAAAIEXKpgJGlqfrSMLdwLM3CjKVJBFWvAOaUDGebiTwOFtQ"
SSH key is good.
fw (manager) # next
fw (admin) # end
fw # exit
Connection to 10.16.2.254 closed.

    Post Implementation Review

    To test your connection just exit out of the old one and initiate a new one with ssh -i $HOME/.ssh/zigafrog manager@10.16.2.254

    .ssh % ssh -i $HOME/.ssh/zigafrog manager@10.16.2.254 
Warning: Permanently added '[10.16.2.254]:8022' (ED25519) to the list of known hosts.
fw # exit
Connection to 10.16.2.254 closed.

    By using the -i argument we tell ssh to use the specific key instead of the default ones stated in the man 1 ssh

    -i identity_file
    Selects a file from which the identity (private key) for public key authentication is read.
    The default is ~/.ssh/id_dsa, ~/.ssh/id_ecdsa, ~/.ssh/id_ecdsa_sk, ~/.ssh/id_ed25519,
    ~/.ssh/id_ed25519_sk and ~/.ssh/id_rsa.  Identity files may also be specified on a per-host
    basis in the configuration file.  It is possible to have multiple -i options (and multiple
    identities specified in configuration files).  If no certificates have been explicitly
    specified by the CertificateFile directive, ssh will also try to load certificate
    information from the filename obtained by appending -cert.pub to identity filenames.
